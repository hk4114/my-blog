### mcp-prompt-server
平时用 AI 工具积累了一堆 Prompt，但真要用的时候总是想不起来放哪了，每次还得复制粘贴，颇为麻烦。

此时，可以尝试使用 MCP Prompt Server 这款 MCP 服务器，把我们常用的 Prompt 变成了可调用的工具，告别复制粘贴。

它将每个 Prompt 模板注册为 MCP 工具，支持自然语言调用，还能在 Raycast、Cursor、Windsurf 等工具中通用。

GitHub：github.com/joeseesun/mcp-prompt-server

主要功能：

- 丰富的内置 Prompt 模板，涵盖代码、写作、产品设计等场景；
- 所有 Prompt 自动注册为工具，支持参数化调用；
- 支持热加载，无需重启即可添加新 Prompt；
- 只需添加 YAML 文件即可扩展新功能；
- 适配 Raycast、Cursor、Windsurf 等主流编辑器；
- 自然语言对话即可调用，还能组合多个工具实现复杂工作流。

通过 npm install 安装后配置到 MCP 客户端即可使用。

公积金年报的微妙变化
公积金制度的存在实际上是一种制度性劫贫济富——少数人用多数人的钱购置资产。

居民不贷款vs“最后通牒博弈”
因为动物精神分为两类，乱纪元中求生存（避险），恒纪元中求发展（加杠杆）。
在行为经济学中，有一个“最后通牒博弈”理论——即A、B两个参与者，A是提议者、B是接受者。如果有100元要分配，A可以提议各自分配一半，也可以自己分配多一点，然后B来决定接受不接受。
理论上作为一个理性经济人，不管A给B分配多少，哪怕只分配1元，对于B来说也是完全新增的，所以看起来B应该怎么样都答应。
但实验结果显示，B对A所提出的不公平的分配会予以否决。比如A提议分配给B20%以下，B就会否决。尽管这样B会损失掉“净增20%的利益”，他们也要“阻止”A能轻松获得“80%利益”的不公平行为。
即心理学上的消极互惠。
我们常常在新闻中看到有人拿着id实名举报，尽管这样他（她）本人承担着很多风险和成本，他们也想要让“犯错”的人接受相应惩罚。
话说回来，前面谈到这新增的100元，在真实经济运行中，也并非完全“凭空”创造的，它借债务的名义，还是让B承担了经济成本的。所以B更加不会愿意A在分配上过于偏袒自己。
一旦B发现，就会产生“消极互惠”的念头。
这也就是为什么A会竭力隐藏自己的分配优势和各种数据。
但隐藏本身，就会产生另一种不信任。我们想象在实验中，B不知道总额是不是100元，只被告知自己可以被分配20元，那么B就会产生猜疑。
而A被分配的金额，最终会在“使用”环节被察觉，进而加深猜疑。其影响也不可小觑。比如明明总额是100元，却被猜疑成200元。
显然猜疑的成本有时候会更大。
最好的降低成本的方式，就是，透明。然后，倾向于B。B可以看到，然后就会从“消极”转为“积极”。
这玩意儿说白了确实很简单。也不需要经济学家反复对着每月数据念经。
然后给出各种荒了大唐的“建议”。
毕竟“天地之间有杆秤”。

学好基本功，AI才能真正助你一臂之力
作者：Manda Putra

我最近一直在尝试把AI融入到日常工作中。必须承认，在处理像增删改查（CRUD）、数据解析这类琐碎任务时，AI确实是个神器。你只需要下达指令，它就能帮你搞定，极大地解放了生产力。但软件开发远不止于此，我们还需要整合系统、性能优化、技术研究，这些都需要真正的理解力。

今天在咖啡馆工作时，我旁边坐着一位同行。他也在用AI写代码，这很正常，我也用。但让我忍俊不禁的是，他为了解决一个极其基础的问题，竟然在和AI反复“拉扯”。那个问题是什么呢？仅仅是一个CSS元素的居中，或是一个忘记绑定`onClick`点击事件的低级Bug。

我内心的独白是：“简直可笑，花几分钟学一下CSS的Flexbox布局，不比你这样反复试错问AI快得多吗？”

我当然不会说出口，毕竟萍水相逢，我也不想打击一个年轻工程师的热情。但这情景让我深有感触。AI需要精确的上下文才能给出好答案，你反反复复地对它说“还没居中呢”，其实是在浪费你自己的生命。**如果你从不想去理解代码片段之间是如何协同工作的，那你当初又何必选择软件开发这个行业呢？**

这让我想起了乔纳森·布洛（Jonathan Blow）一直在警告的那个观点——我们行业的知识体系，正在从根基处“崩溃”。

布洛的核心忧虑是：现代软件建立在层层叠叠的抽象之上，如今的开发者，尤其是新人，越来越依赖顶层的框架和工具（现在又加上了AI），却对底层原理一窍不通。他们知道“怎样”让东西跑起来，却不知道“为什么”能跑起来。

当整个行业都建立在“知其然，不知所以然”的基础上时，我们创造的系统会变得异常脆弱。就像今天这位小哥，他遇到的只是个CSS问题。但如果未来遇到的一个底层性能瓶颈或安全漏洞呢？当没有人再理解事物的本质时，我们又该如何去修复和创造真正可靠的东西？

这正是“崩溃”的真实写照——它不是什么末日天启，而是我们作为一个整体，正在逐渐丧失对复杂技术的掌控力。

AI的出现加剧了这种割裂：

* 对于经验丰富的开发者，AI是如虎添翼的杠杆，能将他们从重复劳动中解放出来，专注于更复杂的架构和创造。
* 对于新手而言，AI却成了一个“黑箱”或“许愿机”，让他们跳过了最关键的学习和试错过程，沦为只会提问的“提示工程师”，根基愈发不稳。

这是一个多么吊诡的世界。一方面技术在飞速进步，另一方面，从业者的平均基本功却在退化。

真心希望那位小哥能看到这篇文章，并明白，真正的成长，源于求知本身。

> 我去，这个翻译完全看不出来翻译痕迹，现在的翻译已经如此自然了嘛

这里一方面当然是因为模型能力越来越强了，一方面也有提示工程（不是提示词）的功劳。

首先我现在最常用的翻译模型是 Gemini 2.5 Pro，表达最自然上下文长度最长，其次才是 GPT-4.5 和 o1-pro，但是 GPT-4.5 长度不行，Claude 翻译的结果很生硬。

然后这里我用了一点提示工程，就是先直译，然后结合上下文解释，再最后”重写“，那么就几乎无翻译痕迹了。

其实也很好理解，这就好比现在你要翻译一篇文章，第一遍的时候你只能按照字面意思理解去翻译，这时候你的思维还是英文的思维，只是“翻译”成了中文，然后就有人对于翻译结果去提问，你在回答问题的时候就深入的理解了文章中一些话背后隐藏的含义，也就对文章的理解更深刻了，这时候让你再用中文重写（本质就是重新意译），而且没有给你条条框框要按照英文去翻译，这时候你实际上已经是中文思维，就是把对文章的理解重新用中文输出，就会很自然，没有翻译的痕迹。
